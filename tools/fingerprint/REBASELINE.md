# Re-baselining CCP's Claude Code wire fingerprint

CCP passes Anthropic's subscription OAuth gate by claiming, on the wire, to be a
specific Claude Code CLI release. Every value that release sends — its version
string, the `anthropic-beta` flags, the `x-anthropic-billing-header` cch
checksum, the model catalog, the wire defaults — is pinned in Rust source as a
**fingerprint profile**. When Claude Code is updated on a machine, those pinned
values can drift out of sync with what real Claude Code now sends, and Anthropic
starts rejecting requests. This doc is the procedure to re-sync the pins to the
installed CLI.

**The invariant this protects: for every supported version, CCP must reproduce
that version's wire fingerprint byte-for-byte.** That exactness is the entire
point of the application — an inexact fingerprint is eventually rejected by
Anthropic's subscription OAuth gate. The goal of a re-baseline is not "update
the pins"; it is "produce a new profile that is byte-exact, and *prove* it."
"Close" is a failure.

This is **detect + guide**, not full-auto. Some fingerprint values (the cch
seed, the exact beta-flag string, the stainless versions) cannot be derived
without a live capture of real Claude Code traffic. Detect what drifted, then do
the edits; never guess wire values — a guessed value is an inexact fingerprint,
which is the one thing this application must never ship.

> **Companion docs, same directory — these are the source of truth, this file
> only orchestrates them:**
> - `BASELINE_HEADERS.md` — the recorded baseline (headers, beta flags, stainless
>   versions) for the current pin.
> - `CCH_ALGORITHM.md` — the documented cch derivation (xxh64 seed, sampling).
> - `check_claude_code_drift.py` — the live drift checker (Step 1) **and** the
>   clean-room capture-vector generator (`--emit-vectors`, Step 4.9).
> - `extract_flow.py` — turns a mitmproxy capture into a readable header+body
>   report (Step 3).
> - `vectors/` — committed real-traffic cch fixtures + `vectors/README.md`. The
>   Rust test that consumes them proves CCP's cch matches REAL traffic over a
>   full body shape (not just self-consistency); regenerated by `--emit-vectors`.

## Mental model: a profile is one struct, everything moves together

The header comment in `src/upstream/fingerprint.rs` says it plainly: these
values "must move together when re-baselining against a new Claude Code
release." A `FingerprintProfile` bundles **all** of:

| Field | What it is | Source of truth |
|---|---|---|
| `name` / `aliases` | profile id, e.g. `cc-2.1.154-sdk-cli`, alias `2.1.154` | `claude --version` |
| `claude_cli_version` | exact version string sent on the wire | `claude --version` |
| `stainless_package_version` / `stainless_runtime_version` | SDK + runtime versions in headers | **live capture** |
| `entrypoint` | `sdk-cli` (how CCP drives it) | stable, rarely changes |
| `beta_reply` + `model_beta_overrides` | the `anthropic-beta` flag string(s) | **live capture** |
| `system_preamble` | canonical Claude Code identity block | `CLAUDE_CODE_SYSTEM_PREAMBLE`, rarely changes |
| `models` | the model **catalog** (canonical ids, context windows) | `claude` model list / release notes |
| `default_model` | which model bare requests resolve to | live capture / release notes |
| `preserve_explicit_model` (bool) | whether an explicit `claude-…` id is forwarded verbatim vs canonicalized; **varies per profile** (false for 2.1.142/2.1.150, true for 2.1.154) | release behavior — set deliberately, don't let it default to `false` |
| `wire_defaults` + `model_wire_overrides` | `max_tokens`, `opus_max_tokens`, `temperature`, `output_effort` | **live capture** |
| `billing` — **two separate mechanisms** | (1) the **cch body checksum**: `xxh64(body, CCH_XXH64_SEED) & 0xfffff`, seed `CCH_XXH64_SEED`. (2) the **billing-suffix**: a sha256-of-UTF-16 sampler with its own `BILLING_SUFFIX_SEED_V1` + `BILLING_SUFFIX_INDICES_V1`. Different seeds, different jobs — do not conflate. | `CCH_ALGORITHM.md` + live cch check |

"Copy the version, the checksums, and the models" is the headline, but a real
re-baseline touches the whole struct. The drift detector tells you which fields
actually changed, so you only edit what moved.

## Files you will touch

- `src/upstream/fingerprint.rs` — the profile structs, `FINGERPRINT_PROFILES`
  registry, `DEFAULT_PROFILE_NAME`, and the version-tagged constants you must add:
  `BETA_CC_2_1_<new>_DEFAULT` + per-model `BETA_CC_2_1_<new>_SONNET`/`_HAIKU`,
  `MODEL_BETA_OVERRIDES_CC_2_1_<new>`, **`WIRE_DEFAULTS_CC_2_1_<new>`**,
  **`MODEL_WIRE_OVERRIDES_CC_2_1_<new>`**, and (only if the cch moved) the cch
  constants. The big one.
- `src/models.rs` — the `CATALOG_CC_2_1_xxx` model catalog. Each `ModelDef` has
  **five** fields, all of which a rename can touch: `canonical`, `cli_name`,
  `aliases`, `context_window`, `max_tokens`. "Copy the models" = add a new catalog
  and update every field that changed — missing `cli_name`/`aliases` on a rename
  silently breaks alias resolution (e.g. `opus` stops resolving).
- `tools/fingerprint/check_claude_code_drift.py` — its own `PINNED_VERSION` /
  `PINNED_PROFILE` constants must be bumped too, or the checker keeps comparing
  against the old version.
- **Test fixtures** that pin the current identity: the profile/cch/suffix
  assertions in `src/upstream/fingerprint.rs` tests and the `DEFAULT_PROFILE`
  dict + model canonicals in `tests/test_integration.py` (see Step 4.7) — a
  `DEFAULT_PROFILE_NAME` bump fails these until updated.
- `tools/fingerprint/BASELINE_HEADERS.md` — the recorded baseline the next
  re-baseline reads first; refresh it (Step 4.8) so it doesn't carry stale values.

## Step 1 — Detect drift (always start here)

Run the live drift checker. It launches a local capture server, drives the
**installed** `claude` CLI against it, and validates the cch checksum its
traffic produces against CCP's pinned algorithm. It also compares installed vs
pinned version.

```sh
uv run tools/fingerprint/check_claude_code_drift.py
```

(Requires a working `claude` on PATH. `uv` pulls the inline `xxhash` dep.)

Read the JSON report:

- `"version_status": "match"` + `"status": "ok"` → **no drift.** The installed
  CLI matches the pin and its cch still validates. Stop — nothing to do.
- `"version_status": "drift"` → installed `claude --version` ≠ `PINNED_VERSION`.
  A new baseline is needed. Note `installed_version`.
- `"status": "drift"` with a `cch mismatch` / `cch stayed at sentinel` detail →
  the **checksum algorithm itself** changed (rare, significant). The cch is no
  longer `xxh64(body, CCH_XXH64_SEED) & 0xfffff`; you cannot just bump the
  version — re-derive the algorithm from a capture (Step 3) per
  `CCH_ALGORITHM.md`, then update **`CCH_XXH64_SEED`** in `fingerprint.rs` **and**
  the matching **`CCH_SEED`** constant in `check_claude_code_drift.py` (the Python
  checker keeps its own copy — if you change one and not the other, the checker
  validates against a stale seed and lies). If the body-hash *structure* changed
  (not just the seed), update the `BillingScheme` / `claude_code_cch_checksum`
  logic too. This is distinct from the billing-**suffix** (`BILLING_SUFFIX_SEED_V1`
  / `BILLING_SUFFIX_INDICES_V1`) — a separate mechanism; check whether *it* drifted
  independently.

Also grab the installed version directly so you have the exact string:

```sh
claude --version    # e.g. "2.1.158 (Claude Code)" → version is 2.1.158
```

Gate "done" on **`status: ok`**, not on `version_status` alone. The checker sets
`status: ok` only when the version matches AND the cch validates, so
`version_status: match` can still come back with `status: drift` — that means the
version didn't change but the cch regressed (stuck sentinel, or the algorithm
moved without a version bump). Do **not** stop on `version_status: match` if
`status` is `drift`; follow the cch-mismatch path above. You are done only when
both `version_status: match` and `status: ok`.

## Step 2 — Confirm what actually drifted

Don't blindly rewrite the whole profile. Diff the installed CLI against the
current pin, field by field, before editing:

1. **Version string** — `claude --version` vs `claude_cli_version` /
   `DEFAULT_PROFILE_NAME` in `fingerprint.rs`. Almost always the trigger.
2. **cch algorithm + billing-suffix** — Step 1's checker validates the cch body
   hash (`xxh64` + `CCH_XXH64_SEED`). The cch *algorithm* is keyed to the CC
   version, not to the model or tool surface, so an `ok` means the algorithm and
   seed are unchanged and you don't touch the cch logic. But understand the
   checker's scope: it computes the cch over **one synthetic body** (a Haiku,
   no-tools "Say OK" probe). It does not prove the cch matches real traffic for
   other models or for tool-bearing bodies — it proves the *function* still
   reproduces what real CC emits for that one body. Don't read `ok` as
   per-model/per-tool validation. The checker also does **not** look at the
   billing-**suffix** (`BILLING_SUFFIX_SEED_V1` / `BILLING_SUFFIX_INDICES_V1`) at
   all — confirm from the Step 3 capture that the suffix is unchanged; both it and
   the cch feed `x-anthropic-billing-header`.
3. **Model catalog** — compare `CATALOG_CC_2_1_<old>` in `src/models.rs` against
   the models the new release exposes (canonical ids + context windows). A
   release may add/rename a model (e.g. opus-4-7 → opus-4-8).
4. **Beta flags / stainless / wire defaults** — these only come from a **live
   capture** (Step 3). With no capture you cannot know whether they changed;
   carry the previous values forward and flag that as an assumption.

## Step 3 — Capture a fresh baseline (REQUIRED for any version bump)

This step is the source of truth for the wire fields the drift checker cannot
see — `anthropic-beta` string(s), stainless package/runtime versions, wire
defaults (`max_tokens`/`temperature`/`output_effort`), and the exact body bytes.
The checker validates the cch and version; these other wire fields are verified
*here or nowhere*. So do this on **every** re-baseline, not just when the cch
algorithm changed — otherwise they're guesses, and a guess is an inexact
fingerprint.

Two fields need a bit more than a single capture:

- **`default_model`** — only observable from a request made with **no** `--model`
  flag (the body's `model` field shows what bare requests resolve to). The
  multi-call recipe below includes that case. Don't read it off a `--model …`
  call — that just echoes the flag.
- **The model catalog** (`CATALOG_CC_2_1_<new>` in `src/models.rs`: each
  `canonical` id + `context_window`) — a single completion body names only the
  one model you called, so the catalog is **not** fully derivable from one
  capture. To learn the exact model ids the *installed* CLI accepts (you need
  these to write the per-model capture calls and the catalog) before driving the
  POST captures, get them from the release/CLI directly — the model-enumeration
  GET that Claude Code issues on startup (inspect it in the same mitmproxy flow),
  and/or the release notes. That GET is not the cch-bearing request (the POST
  filter below excludes it) but it is where the id list shows up. Cross-check each
  model you then capture against it. Capture one call per catalog/override model (recipe below) so
  every pinned model has at least one real-traffic confirmation; treat any model you
  couldn't capture as a flagged assumption.

Use the existing tooling in this directory — don't reinvent it. Read
`BASELINE_HEADERS.md` first: it records the current baseline AND contains the
exact capture command (reproduced below). Read `CCH_ALGORITHM.md` before touching
any cch logic.

**Easiest path — the `capture_baseline.sh` helper.** It does the mitmproxy
intercept method below for you: picks a free port, starts the recorder, drives
the real `claude` CLI for the default model + each model id you pass, and prints
the extracted report. Run it with the models you intend to pin:

```sh
tools/fingerprint/capture_baseline.sh claude-haiku-4-5 claude-sonnet-4-6 claude-opus-4-8
```

It writes the raw flow to `/tmp/cc-baseline.flow` (which contains a live bearer
token — **delete it when done**, never commit it) and the report to
`/tmp/cc-baseline-extract.md`. The manual steps below are the same thing spelled
out, for when you need to vary the capture.

There are two capture methods; they are **not** interchangeable — pick by what
you need:

- **mitmproxy intercept → `.flow` file** (use this for a full re-baseline). You
  need a real `.flow` to run `extract_flow.py`. Run a recording proxy in front of
  the real API and point Claude Code at it, e.g.:

  ```sh
  # terminal 1: record everything to a flow file (reverse-proxy to the real API)
  mitmdump -w /tmp/cc.flow --mode reverse:https://api.anthropic.com --listen-port 8080

  # terminal 2: drive REAL Claude Code through it. Export the env first (works in
  # bash AND zsh — do NOT use `env $base …`: zsh doesn't word-split an unquoted
  # var, so it'd pass one space-laden arg and fail). Match the drift checker's
  # env (disable nonessential traffic + session persistence) to keep the flow
  # clean. Capture ONE call per model you pin, AND one with no --model so the
  # body's "model" field reveals the default resolution.
  #
  # ⚠ Replace the --model ids below with the NEW release's catalog ids — these
  # are the current 2.1.154 ids and will be wrong if the release renamed a model.
  export ANTHROPIC_BASE_URL=http://127.0.0.1:8080
  export CLAUDE_CODE_ENTRYPOINT=sdk-cli
  export CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1
  claude --print --no-session-persistence -- "Say OK"                                # default model
  claude --print --no-session-persistence --model claude-haiku-4-5  -- "Say OK"      # ┐ each catalog/
  claude --print --no-session-persistence --model claude-sonnet-4-6 -- "Say OK"      # ┤ override model
  claude --print --no-session-persistence --model claude-opus-4-8   -- "Say OK"      # ┘ you intend to pin
  ```

  (Omit `--tools ""`: it makes Claude Code send a non-standard empty tool surface
  — fine for reading headers/beta/stainless/wire-defaults, which are
  tool-independent, but don't treat that body as representative of tool-bearing
  traffic.)

  Then extract. `extract_flow.py` does `from mitmproxy.io import FlowReader` and
  has **no** inline uv-deps block, so bare `python extract_flow.py` fails with
  ImportError. Run it under `uv tool run --from mitmproxy`, which provisions an
  ephemeral env that has mitmproxy and then runs `python3` *inside* that env — so
  the import resolves. (This is the same form `BASELINE_HEADERS.md` uses, just
  pointed at your own `.flow` instead of the bundled scenario; verified working.)

  ```sh
  uv tool run --from mitmproxy python3 tools/fingerprint/extract_flow.py \
    /tmp/cc.flow --body-bytes 4000
  ```

  Pick the flow record whose path is `POST /v1/messages` and whose body has both
  `messages` and `system` (that is the real completion call — auth/health probes
  and the model-list GET are not it). `scenarios/01-plain-text.flow` is an example.

- **local fake-server probe** (what the drift checker does; good for cch/version
  only). Point `ANTHROPIC_BASE_URL` at a local HTTP server you control and read
  the request it receives. This does **not** produce a `.flow` file, so
  `extract_flow.py` does not apply — you inspect the captured request directly.
  Sufficient to read headers + body for the cch, but use the mitmproxy method when
  you need the authoritative full header set.

Read the extracted headers/body and record the version-tagged fields into the new
profile (Step 4). If a field genuinely cannot be captured, carry the previous
value forward **and flag it as an unverified assumption** in the commit — never
silently guess.

## Step 4 — Add the new profile (edit source)

> Edit note: `src/upstream/fingerprint.rs` and `src/models.rs` are
> **tab-indented**. Block edits that match on leading whitespace must use real
> tabs — a byte-exact Python edit is the reliable path if a normal edit won't
> apply.

In `src/upstream/fingerprint.rs`, **add a new profile; do not mutate an existing
one** — CCP intentionally keeps the last few versions registered for back-compat
(currently 2.1.154 / 2.1.150 / 2.1.142). Mirror `PROFILE_CLAUDE_2_1_154_SDK_CLI`:

1. Add version-tagged constants for the new release: `BETA_CC_2_1_<new>_DEFAULT`
   **plus the per-model beta variants** the current profile uses
   (`BETA_CC_2_1_154_SONNET`, `BETA_CC_2_1_154_HAIKU` today — make `<new>` copies),
   then `MODEL_BETA_OVERRIDES_CC_2_1_<new>` referencing *those* new constants,
   `WIRE_DEFAULTS_CC_2_1_<new>`, and `MODEL_WIRE_OVERRIDES_CC_2_1_<new>`. Beta
   flags are often per-model — if `MODEL_BETA_OVERRIDES_CC_2_1_<new>` keeps
   pointing at the *old* `BETA_CC_2_1_154_*` constants, the new profile silently
   ships stale per-model beta strings (a model-specific OAuth-rejection vector).
   Carry forward unchanged values; update the ones the capture showed changed.
2. Add `CATALOG_CC_2_1_<new>` in `src/models.rs` (copy the prior catalog, apply
   the release's add/rename/window changes). For each `ModelDef` check **all five
   fields** — `canonical`, `cli_name`, `aliases`, `context_window`, `max_tokens` —
   not just the canonical id; a rename that updates `canonical` but leaves stale
   `cli_name`/`aliases` breaks resolution.
3. Define `PROFILE_CLAUDE_2_1_<new>_SDK_CLI: FingerprintProfile { ... }` wiring
   those constants together.
4. Point the default at it:
   - `pub const DEFAULT_PROFILE_NAME: &str = "cc-2.1.<new>-sdk-cli";`
   - prepend the new profile to `FINGERPRINT_PROFILES` (newest first).
   - keep `LATEST_PROFILE_ALIAS = "latest"` — it resolves to `DEFAULT_PROFILE_NAME`,
     so `latest` now tracks the new release automatically.
5. Only if the cch changed (Step 1 said mismatch): update `CCH_XXH64_SEED` (and
   the `claude_code_cch_checksum` / `BillingScheme` logic if the structure moved)
   per `CCH_ALGORITHM.md`, and check the billing-**suffix** constants separately.
   If Step 1 said `ok`, reuse `BILLING_SCHEME_V1_CCH_XXH64_BODY` unchanged.
6. Update the drift comment at the top of `fingerprint.rs` ("captured <date>
   against claude CLI <version>") and the `finalize_body_bytes` comment listing
   which versions share the cch behavior.
7. **Update the test fixtures that pin the old identity, or `cargo test` fails in
   Step 5 on assertions unrelated to real correctness** — don't mistake those
   cascade failures for a broken profile, and don't skip them as noise (a skipped
   assertion can hide a real cch error). The known hardcoded sites today:
   - `src/upstream/fingerprint.rs` tests: `default_profile_matches_refreshed_claude_code_baseline`
     (asserts `profile.name`/`claude_cli_version`/stainless/model canonicals),
     `profile_registry_resolves_known_selectors` (asserts `resolve_profile("latest").name`),
     the cch snapshot test (`ccp_serialized_body_cch_snapshot_stays_stable`) asserting a
     literal `cch=…`, and `billing_suffix_matches_claude_code_probe` asserting the `"Say OK"`
     version suffix + full default billing-header literal — all keyed to the *default*
     profile, so a `DEFAULT_PROFILE_NAME` bump trips them. Re-derive the new expected
     `cch`/suffix from the rebuilt code, don't hand-edit blindly.
   - **Add a by-name per-model beta test for the NEW profile** (e.g.
     `claude_2_1_<new>_uses_captured_beta_list_per_model`), mirroring the existing
     154/158/161 ones, asserting the full `anthropic-beta` string for all six pinned
     model ids. Aliasing the new beta constants to the prior version's is fine, but the
     test must lock the *resolved* per-model header so a future repoint is caught.
   - `src/routes/completions_v2.rs`: `outbound_build_has_exactly_one_billing_cch_marker`
     also hardcodes the `cc_version=<ver>.<suffix>` prefix — easy to miss because it's
     in a different file than the rest. `cargo test` catches it; don't dismiss it as noise.
   - `tests/test_integration.py`: the `DEFAULT_PROFILE` dict (`name`/`version`/
     `say_ok_suffix`) and the per-model canonical assertions.
8. Update `tools/fingerprint/BASELINE_HEADERS.md` — it is the recorded baseline the
   *next* re-baseline reads first (Step 3 starts there). Refresh its active-baseline
   row, beta-flag strings, stainless versions, and body notes from this capture, or
   the next engineer inherits stale 2.1.154 reference values.
9. **Regenerate the real-traffic cch vectors** so the recovered-capture test validates
   the NEW version (not just self-consistency). Run:
   ```sh
   uv run tools/fingerprint/check_claude_code_drift.py --emit-vectors tools/fingerprint/vectors
   ```
   This drives the installed CLI in a clean room (credential-only temp HOME + clean CWD),
   normalizes the volatile/identifying fields, and writes deterministic
   `vector-<new>-<model>.json` files (see `tools/fingerprint/vectors/README.md`). Then
   point `cch_matches_real_2_1_161_clean_room_capture_vectors`'s `include_str!` paths and
   `<version>` at the new files (or add a new sibling test for the new version, retaining
   the old). This is the test that proves CCP's serialization + cch match REAL traffic for
   the new version's full body shape — the live suite's 200 does not.

Then bump the drift checker's own pins in `check_claude_code_drift.py` — and, if
the cch seed changed in step 5, its `CCH_SEED` too (it keeps its own copy):

```python
PINNED_VERSION = "2.1.<new>"
PINNED_PROFILE = "cc-2.1.<new>-sdk-cli"
# CCH_SEED = 0x...   # only if the cch xxh64 seed changed; must match CCH_XXH64_SEED
```

**Then rebuild before verifying:** `cargo build --release`. If a CCP server is
already running (e.g. the one on :18321), restart it — a running instance keeps
serving the *old* profile from the old binary, so any manual check against it
would silently pass on stale values. (`./tests/run.sh` builds its own binary, so
the gate in Step 5 is unaffected, but manual probes are not.)

## Step 5 — Verify, and understand exactly what each check proves

Be precise about what byte-exactness this step actually establishes — the
procedure's whole value collapses if you over-trust a weak signal. **No single
automated check here proves the *entire* fingerprint is byte-exact.** Exactness
comes from four layers that each cover a different slice:

| Check | What it proves | What it does NOT prove |
|---|---|---|
| `check_claude_code_drift.py` | version string matches; cch body checksum is byte-exact for ONE synthetic body (recomputes `xxh64` with the pinned seed vs. what real Claude Code emits) | nothing about beta flags, stainless versions, wire defaults, or the suffix; only one body shape |
| **Step 3 live capture** | the *authoritative* source for beta flags, stainless versions, wire defaults, default model, suffix — you read them off real traffic and transcribe them | only as good as the capture; a wrong/partial flow → wrong pins |
| **`vectors/` recovered-capture test** (`cch_matches_real_2_1_161_clean_room_capture_vectors`) | CCP's cch reproduces REAL Claude Code traffic over a FULL body shape (`metadata`/`context_management`/`thinking`/`tools`/`cache_control`/multi-block system), per model — proves the serializer + cch agree with real bytes, not just with CCP's own output (non-circular) | exactness of headers/beta/stainless (body-only); only the `--tools ""` shape, not tool-bearing bodies |
| `./tests/run.sh` | Anthropic **currently accepts** the profile end-to-end through CCP | **NOT** byte-exactness. Per `BASELINE_HEADERS.md`, a minimal non-exact request already returns 200 — the gate tolerates *some* drift today. A green run means "accepted now," not "identical to Claude Code." |

So exactness of the un-checked fields rests on **Step 3 being done correctly**,
not on the live suite. Treat `./tests/run.sh` as a necessary acceptance gate (if
it 400s, something is definitely wrong) but **not** as proof of exactness (a green
run does not mean the fingerprint is identical). The earlier the gate tightens on
Anthropic's side, the more a non-exact-but-currently-accepted profile turns into
a production outage — which is why Step 3, not this gate, is where exactness is won.

This codebase has also been burned by a unit test that encoded API-*incorrect*
behavior, so `cargo test` alone is **not** sufficient for anything touching model
resolution, the request body, or the wire fingerprint. Run all gates:

```sh
cargo test                                  # Rust unit tests (incl. fingerprint profile tests)
cargo clippy --all-targets -- -D warnings   # warnings are blockers
uv run tools/fingerprint/check_claude_code_drift.py   # should now report version_status: match, status: ok
./tests/run.sh                              # LIVE integration — calls Anthropic THROUGH CCP
```

> **`./tests/run.sh` prerequisite:** it makes real completion calls through CCP,
> so it needs valid local Claude OAuth credentials at `~/.claude/.credentials.json`
> (a working Claude Max login). Without them the live suite cannot run — and you
> must **not** fall back to `cargo test` as a substitute and call the re-baseline
> done. If you can't run the live suite, say so explicitly and treat the profile
> as unverified-against-Anthropic until someone with creds runs it.

If the re-baseline is wrong in a way the gate *can* see (bad cch, malformed body,
stale/invalid model id), Anthropic 400s in `./tests/run.sh` and it fails — catch
those here. The drift checker should report `version_status: match`, `status: ok`.

## Success criteria

- `claude --version` == pinned `claude_cli_version` == `check_claude_code_drift.py`
  `PINNED_VERSION`.
- Drift checker reports `version_status: match` and `status: ok` (version + cch exact).
- Beta flags, stainless versions, wire defaults, and the suffix were **read from a
  Step 3 live capture** (not carried forward as a guess) — or any carried-forward
  field is explicitly flagged as an unverified assumption.
- New profile registered; `latest` resolves to it; old profiles retained.
- `preserve_explicit_model` set deliberately on the new profile (not defaulted).
- Model catalog matches the release (all five `ModelDef` fields, not just `canonical`).
- Test fixtures (`fingerprint.rs` tests, `tests/test_integration.py` `DEFAULT_PROFILE`,
  the `completions_v2.rs` billing-prefix assertion) updated to the new identity;
  `BASELINE_HEADERS.md` refreshed from this capture.
- A by-name per-model beta test exists for the new profile (full `anthropic-beta`
  string locked for all six pinned model ids).
- Real-traffic cch vectors regenerated for the new version (`--emit-vectors`), the
  recovered-capture test points at them, and it passes (proves cch matches real
  traffic over a full body shape — the non-circular check).
- `cargo test` green, `clippy` clean, **`./tests/run.sh` green** (Anthropic accepts
  the profile — acceptance, not proof of byte-exactness).

## Don't

- Don't mutate an existing pinned profile in place — add a new one.
- Don't set `default_model`, beta flags, or wire defaults to values you haven't
  seen in a real capture — carry forward and flag the assumption.
- Don't trust `cargo test` alone, and don't treat a green `./tests/run.sh` as
  proof of byte-exactness — it proves Anthropic *currently accepts* the profile,
  not that it's identical to Claude Code. Exactness of the un-checked fields is
  won in Step 3 (the live capture), not at the gate.
- Don't re-derive the cch / xxh64 / UTF-16-surrogate math from scratch — it's
  verified-correct and documented in `CCH_ALGORITHM.md`; only touch
  `CCH_XXH64_SEED` / `claude_code_cch_checksum` if the live checker proves the cch
  moved, and check the separate suffix constants
  (`BILLING_SUFFIX_SEED_V1` / `BILLING_SUFFIX_INDICES_V1`) independently.
- Don't change the cch seed in only one place: `CCH_XXH64_SEED` (Rust) and
  `CCH_SEED` (Python checker) must stay equal, or the checker validates against a
  stale seed and reports a false `ok`.
- Don't forget to bump `PINNED_VERSION` / `PINNED_PROFILE` in the drift checker,
  or it silently keeps comparing against the old release.
